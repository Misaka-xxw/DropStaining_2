# Stainer 项目 GPT 与 Codex 协作规范

## 1. 分工

### GPT
- 维护项目事实基线：优先以 `PROJECT_CONTEXT.md`、`TODO.md`、已提交代码、已验证结果和当前 Git 状态为准。
- 判断任务是否具备前置条件，拆分为可验证的小阶段，并编写边界明确的 Codex Prompt。
- 区分已验证事实、推测、Mock 结果、真实设备结果和待现场确认项。
- 审核 Codex 报告：范围、架构、测试、数据库、硬件安全、文档与 Git 状态。
- 只有用户提供实际 `git status --short` 后，才给出精确提交建议。

### Codex
- 仅在 Prompt 明确范围内实现、测试、诊断并报告。
- 遇到环境缺失、超时、未知协议或现场参数时如实报告，不猜测、不降级、不伪造通过。
- 默认不提交 Git，不安装依赖，不连接真实硬件，不扩大任务范围。
- 交付可复核的修改清单、验证证据、遗留风险和 Git 状态。

### 人工确认者
- 确认任务范围、真实设备是否可操作、现场参数是否可信，以及是否接受并提交修改。

---

## 2. 每轮开发流程

```text
事实基线 → Prompt → Codex 实现/验证 → GPT 审核 → Git 检查 → 提交 → 更新项目文档
```

1. 先核对 `PROJECT_CONTEXT.md`、`TODO.md`、本指南和专项文档，以及当前 Git 工作区状态。
2. GPT 只选择前置条件已满足的下一项任务。
3. Codex 执行后不得自动提交。
4. 用户提供 Codex 报告、关键命令结果和 `git status --short`。
5. GPT 判断：通过、需最小返工、或因环境/现场条件未满足而暂停。
6. 提交后同步更新 `PROJECT_CONTEXT.md`、`TODO.md` 和必要专项文档。

---

## 3. Prompt 编写规范

每个 Prompt 至少包含：

1. 任务编号与目标。
2. 当前事实与前置条件。
3. 允许修改的目录、项目和模块。
4. 明确禁止修改或禁止执行的事项。
5. 关键实现要求与安全边界。
6. 构建、测试、诊断和 Git 检查要求。
7. 最终报告格式。
8. 是否允许提交 Git；默认写明“不要提交 Git”。

Prompt 应具体写明，而不是使用“不要乱改”“谨慎处理”等模糊表述。高风险任务必须明确：是否允许数据库迁移、依赖安装、真实端口、SDK 加载、设备连接、硬件动作、外部网络访问和清理数据。

任务应小而可验收。一个任务只解决一个明确阶段，不把“顺手优化”“重构全部相关代码”纳入范围。

---

## 4. 修改代码时的长期原则

- 正式业务权威只能是 ASP.NET Core 应用服务、EF Core 和 SQLite；Mock、浏览器内存、旧原型或测试存储不得成为权威。
- Real 模式必须 fail-closed。未知状态、未校准坐标、未确认安全 IO、未验证协议或通讯异常时不得放行真实动作。
- 适配器不得直接写业务数据库；真实设备命令必须由调用方记录命令生命周期、审计和结构化结果。
- 不得让运行使用“最新配置”替代已冻结的工作流、坐标或液类快照。
- 坐标冲突时，以当前确认的数字孪生 CSV 基线为准；旧文档仅作历史参考。
- 不得无授权修改 `Stainer.sln`、目标框架、平台目标、数据库实体、Migration、运行流程、坐标模型或安全门禁。
- 保持最小 Diff：不无关格式化、不批量换行、不改变文件编码；保护用户已有未提交变更。
- 外部依赖、厂商 SDK、Demo、驱动、授权文件、真实配置和凭据不得提交 Git。
- 当任务范围独立、未影响主解决方案、已有可重复代码级自检和主项目回归验证通过，但受本机外部环境限制无法完成某项正式构建/部署验证时，可在用户明确同意后提交已验证的独立改动；必须把未验证项记录为后续待办，不得写成已通过，并在下一阶段依赖该能力前补齐。这不是跳过正常构建、测试或安全验证的通用例外。

---

## 5. 测试、诊断与数据保护

- “构建通过”“测试发现完成”“部分测试通过”“测试超时”必须分别报告，不能互相替代。
- 使用 `--no-build` 前必须确认对应构建产物存在且与当前代码一致。
- 测试、浏览器验收、诊断和临时数据库必须隔离于 `data/stainer.db`；不得污染正式数据库、WAL、SHM、备份、审计或运行记录。
- 诊断产物（dump、trx、binlog、日志、临时数据库）写入 `%TEMP%` 或任务专用临时目录，完成后清理，禁止进入仓库。
- 不能将超时、跳过、未运行、环境缺失或手工推断报告为“通过”。
- 改动 API、权限、设备状态、工作流或数据模型时，必须补做相关回归检查；必要时检查浏览器重连与多窗口同步。

---

## 6. 真实设备与敏感信息规则

未经任务明确授权，Codex 不得：

- 扫描、枚举或连接 USB、COM、CAN、网络设备；
- 加载、反射、实例化或调用厂商 SDK；
- 发送回零、移动、探液、吸液、加液、加热、制冷、PWM、混合或扫码触发命令；
- 自动重试可能产生物理副作用的命令；
- 安装驱动、Targeting Pack、Build Tools 或下载外部依赖。

真实设备任务必须有：现场许可、安全状态确认、明确协议/NodeID/映射、受控测试范围、动作后回读、超时/断连处理、审计和人工恢复策略。

不得在代码、文档、日志、IPC 响应或提交中暴露真实用户名、SID、SDK 路径、COM、IP、NodeID、坐标、设备序列号、LIS 凭据、患者/样本信息或其他敏感数据。临床样本与 LIS 数据仅使用脱敏测试数据。

---

## 7. Codex 最终报告格式

每轮报告至少包括：

1. 修改文件清单及每项作用。
2. 未修改但重点核查的范围。
3. 构建、测试、静态检查和 Git 检查的实际命令与结果。
4. 数据库、API、配置、硬件、依赖和文档影响。
5. 未完成、未验证、环境受限或风险项。
6. 临时文件和进程清理结果。
7. `git diff --check` 与 `git status --short` 结果。
8. 是否建议提交，以及原因。

报告必须基于实际命令输出；无法验证时明确写“未验证”，不能使用推测性成功结论。

---

## 8. Code Review 与返工判断

GPT 审核时检查：

- 是否严格完成原目标，是否范围漂移；
- 是否修改了未授权模块、数据库、主项目、依赖或安全门禁；
- 是否把 Mock/离线结果误写为真实设备能力；
- 是否有敏感信息、厂商文件、临时产物或本机配置进入 Git；
- 是否测试真正完成，诊断是否隔离并清理；
- 是否更新必要文档与待办。

出现以下任一情况，应返工或暂停，不应提交：

- 关键验收未执行、超时未定位、测试失败未解释；
- 真实设备/SDK/端口操作超出授权；
- 发现硬编码现场参数、凭据、身份信息或路径；
- 数据库、坐标、安全门禁、运行快照或设备命令语义被无授权破坏；
- Git 待提交文件与任务范围不一致。

---

## 9. Git 提交规则

- 默认不提交；完成审核后由用户执行提交。
- 提交前必须运行：

```cmd
git diff --check
git status --short
```

- 只 `git add` 已确认的文件；发现额外文件先解释，不得一并提交。
- 禁止使用破坏性 Git 操作（如强制 reset、清空工作区、重写历史），除非用户明确授权并已说明影响。
- 提交信息应准确描述实际变更，不夸大为“真实设备已支持”或“全部完成”。
